# Задание 1.
В представленном коде можно выделить несколько нарушений принципов SOLID:

## Single Responsibility Principle (SRP):

Класс <b><i>Data</b></i> должен выполнять только одну задачу, но он отвечает как за хранение данных, так и за форматирование их в различные представления. Рекомендуется разделить ответственность по формату на отдельные классы или, по крайней мере, выделить методы форматирования в отдельный класс.

## Open/Closed Principle (OCP):

Класс <b><i>Data</b></i> не открыт для расширения. Если потребуется добавить новый формат, например, <b><i>XML</b></i>, придется изменять существующий код. Лучше создать новую структуру, которая поддерживала бы новые форматы без изменения существующих классов.

## Liskov Substitution Principle (LSP):

Метод <b><i>printAs</b></i> может выбрасывать исключения, если формат не совпадает. Использование <b><i>Data</b></i> в коде, где ожидался <b><i>Printable</b></i>, может привести к неожиданным исключениям во время выполнения.

## Interface Segregation Principle (ISP):

Интерфейс <b><i>Printable</b></i> требует, чтобы все реализации поддерживали все три метода <b><i>printAsHTML</b></i>, <b><i>printAsText</b></i> и <b><i>printAsJSON</b></i>, даже если в некоторых случаях класс может нуждаться только в одном формате. Возможно, стоит рассмотреть возможность создания отдельных интерфейсов для каждого формата.

## Dependency Inversion Principle (DIP):

В функции <b><i>saveTo</b></i> используются конкретные реализации класса <b><i>Data</b></i>, а не абстракции. Это делает код менее гибким, поскольку любые изменения в реализации <b><i>Data</b></i> могут повлечь за собой изменения в методах сохранения. Лучше зависеть от абстракций.
